## 📢 프로젝트 소개

**🎯 타겟 사용자**  
여행 일정 짜는 게 막막해서 어디서부터 시작해야 할지 모르거나, 여러 장소 중 어떤 조합이 좋을지 고민하는 사용자들입니다.

**🛑 문제점**  
여행 지역, 기간, 테마를 정해도 추천 장소가 산발적으로 주어져 어떤 흐름으로 일정을 구성해야 할지 막막하고, 결국 복잡해지거나 포기하는 경우가 많습니다.

**💡 해결 방식**  

사용자가 **지역 → 기간 → 테마(맛집·카페·관광지)** 를 선택하면,  <br>
플랫폼이 자동으로 맛집·카페·관광지를 추천하고, 사용자는 원하는 장소를 **구성**하여 보다 쉽고 빠르게 나만의 일정이 완성되도록 돕습니다.

> 여행 일정을 직접 짜는 게 막막한 당신을 위한 플랫폼입니다.  
> 여행은 가고 싶은 지역과 기간, 테마(맛집·카페·관광지)를 정했지만, <br>
> 정작 ‘어디서부터 시작해야 할지’, ‘어떤 장소들을 언제 방문해야 할지’ 막막할 때가 많습니다.  <br>
> 많은 사용자가 장소 추천은 받지만 그것들을 어떻게 조합해 일정을 구성해야 할지 몰라 포기하거나 흐지부지되는 경우가 많습니다. <br>
> 본 서비스는 이러한 문제를 해결하기 위해 설계되었습니다.  <br>
> 사용자가 지역을 선택하고 기간과 테마를 입력하면, 플랫폼이 지도 기반으로 균형 잡힌 맛집·카페·관광지 추천 리스트를 제공하고, <br>
> 이들을 효과적으로 조합해 나만의 여행 일정을 쉽게 구성할 수 있게 돕습니다.  <br>
> 또한 추천된 장소들은 **지도에서 시각화되어 하루의 흐름이 보다 직관적으로 보이며**, <br>
> 사용자는 구성된 일정을 **언제든 수정하거나 삭제 가능**하고, 마음에 든 일정은 **공유 및 열람을 통해 다른 사용자와 아이디어를 나눌 수 있습니다.**  
> 이로써 복잡하고 힘들었던 여행일정 설계 과정을 단순화하고, 더 빠르고 즐거운 준비 경험을 제공합니다.

```
pipeline {
    agent any

    tools {
        gradle 'gradle'
        jdk 'openJDK17'
    }

    environment {
        SOURCE_GITHUB_URL = 'https://github.com/bynmch/Routy-BE.git'
        MANIFESTS_GITHUB_URL = 'https://github.com/bynmch/Routy-CD.git'
        GIT_USERNAME = 'bynmch'
        GIT_EMAIL = 'byunmch@naver.com'
    }

    stages {
        stage('Preparation') {
            steps {
                script {
                    if (isUnix()) {
                        sh 'docker --version'
                    } else {
                        bat 'docker --version'
                    }
                }
            }
        }
        stage('Source Build') {
            steps {
                git branch: 'main', url: "${env.SOURCE_GITHUB_URL}"
                script {
                    withCredentials([file(credentialsId: 'test', variable: 'secretFile')]) {
                        if (isUnix()) {
                            sh 'cp $secretFile ./src/main/resources/application-secret.yml'
                            sh "chmod +x ./gradlew"
                            sh "./gradlew clean build -x test"
                        } else {
                            bat "copy %secretFile% .\\src\\main\\resources\\application-secret.yml"
                            bat "gradlew.bat clean build -x test"
                        }
                    } 
                }
            }
        }
        stage('Container Build and Push') {
            steps {
                script {
                    withCredentials([usernamePassword(credentialsId: 'DOCKERHUB_PASSWORD', usernameVariable: 'DOCKER_USER', passwordVariable: 'DOCKER_PASS')]) {
                        if (isUnix()) {
                            sh "docker build -t ${DOCKER_USER}/routy_back:${currentBuild.number} ."
                            sh "docker build -t ${DOCKER_USER}/routy_back:latest ."
                            sh "docker login -u ${DOCKER_USER} -p ${DOCKER_PASS}"
                            sh "docker push ${DOCKER_USER}/routy_back:${currentBuild.number}"
                            sh "docker push ${DOCKER_USER}/routy_back:latest"
                        } else {
                            bat "docker build -t ${DOCKER_USER}/routy_back:${currentBuild.number} ."
                            bat "docker build -t ${DOCKER_USER}/routy_back:latest ."
                            bat "docker login -u %DOCKER_USER% -p %DOCKER_PASS%"
                            bat "docker push ${DOCKER_USER}/routy_back:${currentBuild.number}"
                            bat "docker push ${DOCKER_USER}/routy_back:latest"
                        }
                    }
                }
            }
        }
        stage('K8S Manifest Update') {
            steps {
                git credentialsId: 'github',
                    url: "${env.MANIFESTS_GITHUB_URL}",
                    branch: 'main'
                
                script { 
                    withCredentials([usernamePassword(credentialsId: 'github', usernameVariable: 'GIT_USER', passwordVariable: 'GIT_PASS')]) {
                        def githubUrl = env.MANIFESTS_GITHUB_URL.replace('https://', '')
                        if (isUnix()) {
                            sh "sed -i '' 's/routy_back:.*\$/routy_back:${currentBuild.number}/g' routy-deploy/backend-deploy.yml"
                            sh "git add routy-deploy/backend-deploy.yml"
                            sh "git config --global user.name '${env.GIT_USERNAME}'"
                            sh "git config --global user.email '${env.GIT_EMAIL}'"
                            sh "git commit -m '[UPDATE] ${currentBuild.number} image versioning'"
                            sh "git push https://${GIT_USER}:${GIT_PASS}@${githubUrl} main"
                        } else {
                            bat "powershell -Command \"(Get-Content routy-deploy/backend-deploy.yml) -replace 'routy_back:.*', 'routy_back:${currentBuild.number}' | Set-Content routy-deploy/backend-deploy.yml\""
                            bat "git add routy-deploy/backend-deploy.yml"
                            bat "git config --global user.name '${env.GIT_USERNAME}'"
                            bat "git config --global user.email '${env.GIT_EMAIL}'"
                            bat "git commit -m \"[UPDATE] ${currentBuild.number} image versioning\""
                            bat "git push https://%GIT_USER%:%GIT_PASS%@${githubUrl} main"
                        }
                    }
                }
            }
        }
    }

    post {
        always {
            script {
                if (isUnix()) {
                    sh 'docker logout'
                } else {
                    bat 'docker logout'
                }
            }
        }
        success {
            echo 'Pipeline succeeded!'
        }
        failure {
            echo 'Pipeline failed!'
        }
    }
}
```

## 🍪 프로젝트 회고

| 팀원 |  내용 |
|--------|-----------|
| 곽동근 | 이번 프로젝트에서는 마이페이지, 일정 수정, 리뷰 작성 기능에 필요한 백엔드 설계와 API구조 설계를 주로 담당하였다. 해당 기능들은 일정, 리뷰, 사용자 정보 등 다향한 데이터가 맞물려 있기 때문에 단순히 하나의 기능만 구현하는것이 아니라 서로 다른 도메인의 흐름을 조합해 구조를 설계하는 작업이 많았다. 마이페이지 특성상 다른 화면과 데이터가 서로 얽혀있어서 조원들이 맡은 기능들과 자연스럽게 겹치는 부분이 많았고, 이러한 부분을 조율하며 전체 흐름을 맞추는데 집중했다. 특히 마이페이지는 일정 목록, 캘린더 반영, 리뷰 여부, 북마크 여부 등 여러 요소가 동시에 표시되는 화면이라서, 이를 지원하기 위한 DTO 구조를 처음부터 직접 설계하고 데이터 형태를 일관되게 유지하기 위해 수차례 수정 작업을 수행했다. 프론트엔드는 원래 다른 팀원이 개발을 담당했지만, 마이페이지 등 관련 API들이 실제 화면에서 기대한 형태로 동작하지 않는 부분이 많아 프론트 코드도 함께 분석하며 수정하는 일이 잦았다. 이 과정에서 FE/BE 간 데이터 흐름을 직접 맞춰가며 문제를 해결했고, 필요할 경우 기능을 새롭게 추가하거나 API 자체를 보완하기도 했다. 이러한 경험을 통해 “기능 하나”를 구현하는 것이 아니라 시스템 전체가 자연스럽게 이어지도록 만드는 것이 개발에서 얼마나 중요한지 깨닫는 계기가 되었고, 백엔드를 거의 설계하다 보니 전반적인 API 구조를 잡아두고 나중에 프론트 쪽 요구사항이 변경될 때 다시 되돌아가야 하는 일이 많았다. 초기 설계가 완벽하지 않았거나 팀 간 소통이 미흡했던 경우도 있어 불필요한 반복 작업이 있었다.이번 프로젝트는 개인적으로 가장 많은 성장을 경험한 프로젝트였다. 데이터 모델링 → DTO 설계 → 매퍼/쿼리 구성 → API → 프론트 연동까지 이어지는 전체 흐름을 직접 연결해보며, 특정 지점에서 발생한 오류를 DB부터 화면까지 전부 추적해 해결하는 능력이 크게 향상되었다. 단순 “코드 작성”을 넘어서, 전체 시스템을 하나의 흐름으로 바라보는 시야를 얻게 된 점이 이번 프로젝트에서 가장 큰 배움이었다고 생각한다. |
